# リファクタリングパターン集

## パターン 1: SKILL.md から references/ への分離

### いつ適用するか

SKILL.md が肥大化している場合（目安: 80行以上）。以下のセクションは references/ に分離する候補:

| 分離候補 | 理由 |
|---------|------|
| テンプレート（markdown コードブロック） | ワークフローの流れを読む際にノイズになる |
| よくある落とし穴・ナレッジ集 | 蓄積型のコンテンツであり、ワークフローとは独立 |
| コミットメッセージフォーマット | 参照頻度が限定的（コミット時のみ） |
| 品質基準・重要度分類 | 詳細な判定基準はワークフロー概要に不要 |
| ナンバリング規則 | 出力フォーマットの詳細であり、分析フローとは独立 |

### SKILL.md に残すべきもの

- frontmatter（name, description）
- 前提メモリセクション
- 鉄則（5項目以内が理想）
- ワークフロー概要（Phase/Step の見出しと簡潔な説明）
- リファレンステーブル

### 分離後の参照方法

SKILL.md 末尾にリファレンステーブルを設ける:

```markdown
## リファレンス

| ファイル | 内容 | 参照タイミング |
|---------|------|--------------|
| `references/templates.md` | レポートテンプレート | Step 4: レポート出力時 |
| `references/pitfalls.md` | よくある落とし穴 | Phase 3: 実装時 |
```

**参照タイミング列は必須**。ワークフローの Phase/Step に対応させることで、スキル実行時に必要なタイミングでのみ読み込まれる。

## パターン 2: 前提メモリの宣言

### いつ適用するか

スキルがプロジェクト固有の情報（パス、コマンド、プロジェクトキーなど）を必要とする場合。

### 標準フォーマット

SKILL.md の冒頭（frontmatter の直後、本文の前）に配置:

```markdown
## 前提メモリ

スキル実行前に以下の Serena メモリを読み込むこと:

| メモリ名 | 用途 |
|---------|------|
| `project/plan_docs_location` | 計画書の保存先パスとコミットルール |
| `project/architecture_knowledge` | 開発コマンド（typecheck, test, lint） |
```

### よく使うメモリの種類

| メモリ名 | 格納する情報 | 参照するスキルの例 |
|---------|------------|------------------|
| `project/plan_docs_location` | 計画書ディレクトリのパス、コミットルール | plan-feature, impl-driver, pr-review |
| `project/architecture_knowledge` | レイヤー構成、開発コマンド、コード変更パターン | plan-feature, impl-driver |
| `project/doc_survey_targets` | ドキュメントサーベイ対象ファイルリスト | plan-doc-update |
| `project/jira_project_key` | JIRA プロジェクトキー | jira-ticket |

## パターン 3: プロジェクト固有情報の汎用化

### いつ適用するか

スキルを複数プロジェクトで再利用したい、または公開リポジトリに置きたい場合。

### 汎用化の手順

1. **プロジェクト固有の記述を特定する** — ファイルパス、エンティティ名、チケット番号、ツール名、ユーザー名
2. **Serena メモリに退避する** — 先にメモリに書き込み、情報が保存されたことを確認
3. **スキルの記述を汎用化する** — 具体例を一般的な表現に置き換える
4. **前提メモリセクションを追加/更新する** — メモリへの参照を宣言

### 汎用化の変換例

| カテゴリ | プロジェクト固有 | 汎用化後 |
|---------|----------------|---------|
| ディレクトリパス | `backend/src/domain/` | 「ドメイン層ディレクトリ」 |
| ドキュメントパス | `docs/docs/` | 「ドキュメントディレクトリ」 |
| 開発コマンド | `docker compose exec -T backend npm run typecheck` | 「プロジェクトの typecheck コマンド」 |
| エンティティ名 | `SignInFlow`, `Invoice` | `Entity`, `モデル` |
| チケット番号 | `AG-354` | `PROJ-001` |
| 状態名 | `AwaitingMfa` | `PendingVerification` |
| ファイル名 | `sign-in-flow.ts` | `auth-flow.ts` |
| ユーザー名 | 個人名 | `user1` |
| ツール名 | 固有ツール名 | `custom-bot` |

### 汎用化しなくてよいもの

- 汎用的な技術用語（Prisma, `$transaction`, DI, OTP など）
- フレームワーク名（Hono, InversifyJS など）
- 一般的なパターン名（co-located pattern, clean architecture など）

## パターン 4: リファレンステーブルの標準化

### 3列構成を標準とする

```markdown
| ファイル | 内容 | 参照タイミング |
|---------|------|--------------|
```

- **ファイル**: `references/xxx.md` 形式
- **内容**: ファイルに含まれる情報の概要（括弧書きで主要トピックを列挙）
- **参照タイミング**: ワークフローの Phase/Step に対応させる

### インライン参照からの移行

散在するインライン参照:
```markdown
判定記号・検証手順の詳細は `references/verification-guide.md` を参照。
```

標準化後:
```markdown
## リファレンス

| ファイル | 内容 | 参照タイミング |
|---------|------|--------------|
| `references/verification-guide.md` | 判定記号・検証手順・分類別の検証ポイント | Step 2-3: 主張の抽出・検証時 |
```

## パターン 5: SKILL.md の理想的な構造

```
---
frontmatter (name, description)
---

# タイトル

## 前提メモリ（Serena メモリを使う場合）

## 鉄則（3-5項目）

## ワークフロー（Phase/Step 概要）

## リファレンス（3列テーブル）
```

目安行数: **50-80行**。100行を超えたら分離を検討する。
