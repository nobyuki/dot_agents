# プロジェクト固有の知識

プロジェクト固有のアーキテクチャ構成・技術スタック・ディレクトリ配置は前提メモリから取得すること。

以下は汎用的なガイドラインを示す。

## アーキテクチャレイヤー

計画書ではクリーンアーキテクチャの各レイヤーに沿って設計詳細を記述する。
既存コードが独自パターンを採用している場合はそちらを優先し、逸脱の理由を考慮事項に記載する。

| レイヤー | 計画書での記載内容 |
|---------|------------------|
| ドメイン | エンティティ変更、バリューオブジェクト、ドメインサービス |
| アプリケーション | ユースケース、DTO、ポートインターフェース |
| インフラストラクチャ | リポジトリ実装、外部サービスアダプタ |
| インターフェース | API ルート、Web UI |
| DI | DI コンテナのシンボル追加、バインディング変更 |
| DB | スキーマ変更、マイグレーション SQL |

## コード変更の記載パターン

- **DB スキーマ**: モデル定義のコードブロック（★追加 コメント付き）
- **エンティティ**: constructor, create(), update() の TypeScript スニペット
- **ユースケース**: 処理フローの擬似コード（重要な分岐・エラー処理を含む）。状態ガードは条件がシンプルになる方を選ぶ
- **API**: エンドポイントパス、リクエスト/レスポンスの JSON サンプル、OpenAPI スキーマ
- **UI**: HTML/JSX スニペット、画面要素の説明

## 画面更新を含む場合

UI の変更を含む計画書では、実装ステップの最後に**手動確認ステップを必ず設ける**。自動テストではカバーできない表示・操作性の問題を検出するため。

手動確認ステップの例:
```
### Step N: 動作確認

- [ ] 対象画面をブラウザで開き、表示が期待通りであること
- [ ] 新規追加した UI 要素が正しく動作すること
- [ ] 既存の UI 要素に意図しない影響がないこと
- [ ] エラー状態・空データ状態の表示を確認
```

## テスト計画の記載

テストファイルはソースファイルと同じディレクトリに配置する（co-located pattern）。

実装ステップの中にテスト項目を含める。以下のパターンをカバーする:
- 正常系（happy path）
- エッジケース（境界値、null/undefined、空配列）
- エラー系（fail-close / fail-open の判断）
- 既存テストへの影響
