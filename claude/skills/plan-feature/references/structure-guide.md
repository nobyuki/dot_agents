# 機能実装計画書 テンプレート詳細

## §1. 背景

### 必須サブセクション

#### 現状の課題
現在のシステムの問題点を具体的に記述する。

#### 方針
課題に対するアプローチを箇条書きで記述する。

#### 現在のフロー / 変更後のフロー（任意）
処理フローの変更がある場合、ASCII ツリー図で before/after を示す。

```
サインイン
  ├─ 条件A
  │   ├─ パターン1 → 結果1
  │   └─ パターン2 → 結果2
  └─ 条件B
      └─ パターン3 → 結果3
```

**核心的な変更** を箇条書きでまとめる（2-3項目）。

## §2. 影響範囲

テーブル形式で記述する:

| コンポーネント | 現状 | 影響 |
|---------------|------|------|
| `Entity名` エンティティ | 現在の状態 | 追加・変更内容 |
| `UseCase名` | 現在のロジック | 変更後のロジック |
| `API名` | 現在のエンドポイント | 追加パラメータ等 |
| `UI名` | 現在のUI状態 | UI変更内容 |
| `Repository名` | 現在の実装 | マッピング変更等 |

## §3. 設計詳細

コンポーネントごとにサブセクション（3.1, 3.2, ...）を設ける。

### コードスニペットの書き方

**Prisma スキーマ変更**:
```prisma
model ModelName {
  existingField  String
  newField       Boolean  @default(false) @map("new_field")  // ★追加
}
```

**エンティティの constructor / create / update**:
```typescript
constructor(props: {
  existingProp: string;
  newProp?: boolean;  // ★追加
}) {
  this.newProp = props.newProp ?? false;
}
```

**ユースケースのロジック変更**:
```typescript
// 変更前のコメント付きで before/after を示す
// Before:
if (oldCondition) { /* ... */ }

// After:
if (newCondition) { /* ... */ }
```

**マイグレーション SQL**:
```sql
ALTER TABLE table_name ADD COLUMN column_name TYPE NOT NULL DEFAULT value;
UPDATE table_name SET column_name = value WHERE condition;
```

### 参照フローの記法

エンティティ間の参照チェーンを示す場合:
```
startId (Source)
  ↓ Repository.findByXxx()
middleEntity
  ↓ Property.relatedId
targetEntity.targetField
```

## §4. 考慮事項

番号付きサブセクション（4.1, 4.2, ...）で記述。以下のカテゴリを検討:

- **後方互換性**: 既存データ・既存APIへの影響
- **マイグレーション戦略**: データ移行の方法と順序
- **エッジケース**: 未設定データ、null ケース、競合状態
- **不採用案**: 検討したが採用しなかった方式とその理由
- **関連機能への影響**: 変更しないが関連する機能の説明
- **既存パターンとの整合性**: クリーンアーキテクチャの原則から逸脱する場合、既存コードとの一貫性を優先した根拠

## §5. 実装ステップ

Step ごとに関連する作業をまとめる。各ステップ内はチェックボックスリスト。

```markdown
### Step 1: [カテゴリ名]

- [ ] タスク1の具体的な説明
- [ ] タスク2の具体的な説明
```

推奨されるステップ順序:
1. DB スキーマ変更 + マイグレーション
2. ドメイン層（エンティティ、バリューオブジェクト）
3. インフラストラクチャ層（リポジトリ、アダプタ）
4. アプリケーション層（ユースケース、DTO）
5. インターフェース層（API、UI）
6. DI コンテナ更新
7. テスト
8. 動作確認（画面更新を含む場合は必須）

### 動作確認ステップ（画面更新時は必須）

UI の変更がある場合、最終ステップに手動確認を含める:

```markdown
### Step N: 動作確認

- [ ] 対象画面をブラウザで開き、表示が期待通りであること
- [ ] 新規追加した UI 要素が正しく動作すること（クリック、入力、送信）
- [ ] 既存の UI 要素に意図しない影響がないこと
- [ ] エラー状態・空データ状態の表示を確認
- [ ] レスポンシブ表示の確認（該当する場合）
```

## 既存計画書の参照

以下のタイプの既存計画書をスタイルの参考にできる:

| タイプ | 特徴 |
|--------|------|
| 標準的な機能追加 | 背景のフロー図、設計詳細のコード、考慮事項、レビュー対応セクションあり |
| 大規模な機能追加 | 複数のUI変更、設計検討事項が充実、動作確認手順が詳細 |
| 複雑なDB設計 | 2層設計などの設計パターン。テーブル設計と出力仕様が詳細 |
| 管理画面追加 | API・UI・ユースケースのバランスが良い |
| リファクタリング/削除系 | フェーズ分けと進捗管理が特徴的 |
