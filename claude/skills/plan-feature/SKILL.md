# 機能実装計画書の作成

## 前提メモリ

スキル実行前に以下の Serena メモリを読み込むこと:

| メモリ名 | 用途 |
|---------|------|
| `project/plan_docs_location` | 計画書の保存先パスとコミットルール |
| `project/architecture_knowledge` | アーキテクチャ構成、開発コマンド、既存計画書の参照 |

プロジェクトで使用する機能実装計画書を作成する。
計画書は計画書ディレクトリに Markdown 形式で保存する。

## 計画書作成の原則

1. **コードを先に読み、計画を後に書く** — 推測で書かない。必ず既存コードを確認してから設計詳細を記述する
2. **before/after を明示する** — 現状の課題と変更後の姿を対比で示す。Before コードは実際のコードと必ず照合する
3. **スコープを明確にする** — 何を変えるか（影響範囲）と何を変えないか（対象外）を区別する
4. **コードスニペットは実装指針** — 最終的なコードそのものではなく、方向性を示す擬似コードとして書く
5. **考慮事項で設計判断を記録する** — 「なぜこうしたか」「なぜこうしなかったか」を残す
6. **進捗管理にチェックボックスを使う** — 実装中に状態を更新できるようにする
7. **既存実装パターン優先** — クリーンアーキテクチャに準拠しつつ、既存コードの一貫性を優先する
8. **コミットハッシュは記載しない** — 計画書はローカル専用であり、ハッシュは時間経過で意味を失う

## ワークフロー

1. **情報収集** — 影響を受けるエンティティ・ユースケース・リポジトリ・API・UI を特定。`find_symbol` / `find_referencing_symbols` で依存関係を追跡
2. **計画書の作成** — テンプレートに従い記述。詳細は `references/structure-guide.md` を参照
3. **ユーザーレビュー** — 背景→影響範囲→設計詳細の順で段階的に提示し、フィードバックを反映
4. **スキルの更新** — 作成過程で得た知見を SKILL.md や references に反映。既存計画書参照テーブル（`references/structure-guide.md` 末尾）に追加

## 計画書の構造（概要）

```
# [チケット番号]: [タイトル]
## 1. 背景（課題・方針・フロー図）
## 2. 影響範囲（コンポーネント別テーブル）
## 3. 設計詳細（コンポーネントごとのサブセクション + コードスニペット）
## 4. 考慮事項（後方互換性、エッジケース、不採用案）
## 5. 実装ステップ（チェックボックス付きタスク）
## 6. 変更ファイル一覧（変更/新規/テスト/削除/参照）
## 7+. スコープ外（任意）
```

## ファイル名の規則

- チケット番号あり: `[チケット番号]-[kebab-case-description].md`（例: `PROJ-001-feature-description.md`）
- チケット番号なし: `[kebab-case-description].md`（例: `feature-description.md`）

## リファレンス

| ファイル | 内容 |
|---------|------|
| `references/structure-guide.md` | テンプレート詳細（各セクションの書き方、コードスニペット例、既存計画書参照） |
| `references/project-knowledge.md` | アーキテクチャレイヤー、コード変更パターン、画面更新・テスト計画の記載ルール |
| `references/checklist.md` | 計画書完成前の品質チェックリスト |
