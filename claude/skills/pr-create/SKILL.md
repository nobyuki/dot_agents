# PR 作成

ブランチの変更内容を分析し、PR のタイトルと本文をユーザーと練り上げてから GitHub に登録するスキル。

## 鉄則

1. **git push は絶対に実行しない** — リモートに push されていない場合はユーザーに依頼する
2. **PR 登録はユーザーの明示的な承認を待つ** — 「作成して」「登録して」と言われるまで `gh pr create` を実行しない
3. **ドラフトを先に `/tmp` に出力する** — いきなり登録せず、PR 案をファイルで提示してレビューを受ける
4. **レビューコメントは妥当性を検証してから反映する** — 指摘を鵜呑みにせず、差分に照合して判定する

## ワークフロー

1. **Phase 1: ブランチ状態の確認** — git log/diff/status でベースブランチからの差分・リモート追跡状態・未コミット変更を並列取得
2. **Phase 2: PR 案の作成** — `/tmp/pr-{branch-short-name}.md` にタイトル・Summary・Changes・Notes・Test plan を出力
3. **Phase 3: レビュー反映** — ユーザーやレビューツールのフィードバックを検証し、妥当な指摘のみ反映（繰り返し）
4. **Phase 4: PR 登録** — ユーザーの承認後に `gh pr create` で登録し、PR URL を返す

各フェーズの詳細手順は `references/workflow.md` を参照。
