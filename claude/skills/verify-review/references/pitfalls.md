# 過去の検証から得た知見

- **「見落とし」と断定する前に設計ドキュメントを確認する**: レビューが言及していない事項を「見落とし」と評価する前に、設計ドキュメントや ADR で意図的な設計判断として記載されていないか確認する。設計方針として明記されている内容をレビューの欠陥として指摘すると、検証自体が不正確になる
- **「問題点」と「改善余地」を使い分ける**: 事実誤認は判定テーブルで `❌ 不正確` として扱う。レビューの観点不足・的外れな提案・網羅性の限界は「改善余地」セクションに記載し、「問題点」と断定しない。表現のトーンが強すぎると、検証レポートの信頼性を損なう
- **段階的更新時の整合性**: レポートを段階的に更新する場合（複数レビューの検証、ユーザーからの訂正反映）、解決済みの事項への参照が残りがちになる。更新のたびにレポート全体の整合性を確認する
- **「バグ」と「要件確認」を区別する**: レビューが「〜できてしまう」と指摘していても、設計ドキュメントでその動作が明示されている場合は欠陥ではなく要件確認レベルの懸念。スコープ判定は「境界上」に寄せて扱い、検証結果にもドキュメントの根拠を明記する。設計ドキュメントを照合せずに欠陥断定すると検証の信頼性を損なう
- **問題発見力の◎/○はレビュアーへの情報公開度で判定する**: PR 本文に明記された懸念をレビュアーが指摘しても、それは公開情報の認識であり○止まり。一方、レビュアーに非公開の設計資料で既知の懸念であっても、レビュアーが独立にコードから特定していれば◎と評価すべき
- **「対応不要」と「このPRでは必須ではない」を区別する**: 指摘が事実として成立する懸念であれば、「対応不要」は結論として強すぎる。「このPRでの必須対応ではないが、フォローアップを推奨」が正確。特にバックポート PR では、オリジナルの設計を変更する指摘はスコープ外寄りだが、懸念自体の有効性は否定しない
- **行番号のずれを過大評価しない**: GitHub レビューでの手打ち行番号が実際のファイルとずれていることは日常的に起こる。主張内容が正しければ「軽微な記載揺れ」として扱い、「レビューの信頼性に関わる」とまで書かない
- **スコープ判定の「境界上」を適切に使う**: PR が新規追加したコードへの指摘は、たとえ設計判断がオリジナル（別PR等）に由来するものであっても、純粋な「スコープ外」とは言い切れない。「境界上（スコープ外寄り）」のように粒度を上げて判定する。一律「スコープ外」とすると指摘の有効性を不当に低く見積もるリスクがある
- **レビューの「観察」と「提案」を分けて判定する**: レビューの観察（現象の指摘）が正確でも、提案（修正方針）の妥当性は別途検証が必要。特に提案が変更箇所の下流に影響する場合、呼び出し先のバリデーションや制約まで追わないと判定できない。観察の正確性だけを見て「対応推奨」とすると、下流で破壊的になる提案を是認するリスクがある
- **問題発見力の評価には提案の実用性も含める**: 論点の発見自体が優れていても、提案が下流制約を見落としていて採用できない場合は◎ではなく○が妥当。「良い問いを立てたが、答えが間違っている」ケースは発見力を割り引いて評価する
- **「到達不能」判定では呼び出し先の状態依存パスまで追う**: 引数が固定値でも、呼び出し先が既存データを返すパス（findOrCreate、キャッシュ再利用等）を持つ場合、既存データの状態次第で「固定値では到達しないはず」の分岐に到達しうる。呼び出し元の引数だけ見てデッドコードと判定せず、呼び出し先で外部状態が介入するパスまで追う
- **レビュアーに非公開の資料を検証の主根拠にしない**: 設計資料の公開状態（リポジトリにコミットされているか、ローカル専用か）を確認する。非公開の場合、検証の裏取りに使うのは問題ないが、レポートの「根拠」欄や「追加対応の要否」で主根拠として前面に出すと、「手元事情を知っていればそう判断できる」に寄り、検証の客観性が弱まる。主根拠はPR差分・コード・コミット履歴などレビュアーも観測可能な事実に置き、非公開資料は補足に留める
- **レビュー対応でコードを変更したらレポート内の行番号を全て更新する**: レビュー対応としてコードを変更すると、レポート内の行番号参照が一斉に stale になる。「段階的更新時の整合性」の一環だが、特にクロス検証テーブルのように複数箇所で同じ行番号を参照している場合に見落としやすい
- **「永続的」等の時間的断定は回復パスまで追う**: レビューが「永続的に喪失」「恒久的にブロック」等と主張している場合、タイムアウト・期限切れ・別フローの完了等の回復パスが存在しないか確認する。回復パスがあれば `⚠️ 部分的に正確`（過剰表現）として扱う。物理削除のような不可逆操作と、それに伴う状態ブロックの可逆性は分けて評価する
- **提案がPRの解消対象を再導入するケースに注意する**: 「観察と提案を分けて判定する」の具体パターン。レビューが「Xすれば解決する」と提案していても、そのXがPR本文やコミットメッセージで「Xをやめた理由」として説明されている場合、提案はPRの主目的を壊す。PR本文・コミットメッセージ・コード内コメントを照合してから提案の妥当性を判定する
- **スコープ判定は一発で決めず反復的に精緻化する**: 初期判定（例: スコープ外）をクロス検証で異論が出たら再検討し、収束点を見つける。典型パターン: 「既存コードの問題 → スコープ外」と判定したが、「PRの変更がその問題を新たな論点として顕在化させた」という観点で「境界上」に修正。初期判定を固守せず、PR の差分が作り出す新しい文脈を考慮する。レポート内では旧判定をログとして残しつつ、最終判定を明記する
- **スコープ判定の変更はレポート全体に波及する**: スコープ判定を「スコープ外」→「境界上」に修正した場合、判定テーブル・総合評価・スコープ集計のすべてを一括更新する。「ラウンド2で修正済み」と書いてもラウンド1の記述が残ったままだと文書全体が矛盾する
- **依存関係の観察と実害リスクは別軸で判定する**: レビューが「A が B に依存しているので C が起きうる」と主張している場合、A → B の依存が事実でも、B が実際に変わりうるかはデータモデルや実装の不変性制約に依存する。依存関係の観察と、その依存が実害につながるかは別軸で判定する。前者だけ検証して後者を省略すると、過剰断定を見逃すか、逆に有効な指摘を却下するリスクがある
- **PR 本文は可変ドキュメントであり、取得時点と投稿時点のずれに注意する**: `gh pr view` で取得できる PR 本文は現在時点のスナップショットであり、レビュー投稿時点の内容とは限らない。author がレビュー対応として PR 本文（テストプラン、設計意図等）を更新することは日常的に起こる。レビューコメントが PR 本文を引用しているとき、以下の手順で時点差を確認する: (1) レビューの `submitted_at` と PR の `updated_at` を比較する。`updated_at` が後なら更新の可能性がある (2) author の返信コメントに「PR本文を更新した」「テストプランを修正した」等の言及がないか確認する (3) 引用が不正確に見える場合、レビュー時点では正確だった可能性を検討する。更新後の本文を根拠にレビューの引用を「誤り」と判定すると、検証自体が不正確になる
- **過去レビューで受容済みの設計を must-fix として再指摘するケースの扱い**: 同一レビュアーが子 PR で approve した設計を、統合 PR で CHANGES_REQUESTED にするパターンがある。過去の approve 履歴（子 PR のレビュー本文・issue comment）を照合し、レポートでは「過去レビューでは受容されていた設計への別案提案」として整理する。人物評（「一貫性がない」等）に寄らず、指摘の性質として記述する。PdM 合意済みの仕様であればその旨も根拠に含める
- **ベースブランチ側の並行変更を把握していないレビュー指摘の扱い**: レビュアーがベースブランチ側で入った変更を把握せず、差分認識を誤るケースがある。検証時はベースブランチ時点の実装を git で確認し（`git show <base-branch>:path/to/file`）、「この PR の差分」と「ベースブランチで既に入っていた変更」を区別する。レポートにはベースブランチ時点の事実を明記する
- **author が「仕様をコードに寄せた」のか「コードを仕様に合わせた」のかを区別する**: レビュー指摘への対応で author が PR 本文やドキュメントを更新した場合、コードを修正して仕様に合わせたのか、それとも仕様記述をコードの現状に寄せたのかで意味が大きく異なる。後者（仕様をコードに寄せた）の場合、特に画面再テスト等の検証を経ていなければ、更新後の記述をもってレビュー指摘を退ける根拠にはしにくい。事後的な説明の書き換えと、検証を伴う設計判断は区別して扱う
