# 過去の検証から得た知見

- **「見落とし」と断定する前に設計ドキュメントを確認する**: レビューが言及していない事項を「見落とし」と評価する前に、`project/plan_docs_location` メモリが示すディレクトリで意図的な設計判断として記載されていないか確認する。設計方針として明記されている内容をレビューの欠陥として指摘すると、検証自体が不正確になる
- **「問題点」と「改善余地」を使い分ける**: 事実誤認は判定テーブルで `❌ 不正確` として扱う。レビューの観点不足・的外れな提案・網羅性の限界は「改善余地」セクションに記載し、「問題点」と断定しない。表現のトーンが強すぎると、検証レポートの信頼性を損なう
- **段階的更新時の整合性**: レポートを段階的に更新する場合（複数レビューの検証、ユーザーからの訂正反映）、解決済みの事項への参照が残りがちになる。更新のたびにレポート全体の整合性を確認する
- **「バグ」と「要件確認」を区別する**: レビューが「〜できてしまう」と指摘していても、`project/plan_docs_location` 配下のドキュメントでその動作が明示されている場合は欠陥ではなく要件確認レベルの懸念。スコープ判定は「境界上」に寄せて扱い（スコープ分類は `references/workflow.md` を参照）、検証結果にもドキュメントの根拠を明記する。設計ドキュメントを照合せずに欠陥断定すると検証の信頼性を損なう
- **問題発見力の◎/○はレビュアーへの情報公開度で判定する**: PR 本文に明記された懸念をレビュアーが指摘しても、それは公開情報の認識であり○止まり。一方、計画書（`project/plan_docs_location`）はレビュアーに非公開のため、計画書で既知の懸念であってもレビュアーが独立にコードから特定していれば◎と評価すべき
- **「対応不要」と「このPRでは必須ではない」を区別する**: 指摘が事実として成立する懸念であれば、「対応不要」は結論として強すぎる。「このPRでの必須対応ではないが、フォローアップを推奨」が正確。特にバックポート PR では、オリジナルの設計を変更する指摘はスコープ外寄りだが、懸念自体の有効性は否定しない
- **行番号のずれを過大評価しない**: GitHub レビューでの手打ち行番号が実際のファイルとずれていることは日常的に起こる。主張内容が正しければ「軽微な記載揺れ」として扱い、「レビューの信頼性に関わる」とまで書かない
- **スコープ判定の「境界上」を適切に使う**: PR が新規追加したコードへの指摘は、たとえ設計判断がオリジナル（別PR等）に由来するものであっても、純粋な「スコープ外」とは言い切れない。「境界上（スコープ外寄り）」のように粒度を上げて判定する。一律「スコープ外」とすると指摘の有効性を不当に低く見積もるリスクがある
