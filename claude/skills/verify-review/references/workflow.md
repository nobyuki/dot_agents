# 検証ワークフロー詳細

## Phase 1: レビューコメントの取得

URL のパターンに応じて `gh api` で取得する。

```
# Review 全体 (pullrequestreview-xxx)
gh api repos/{owner}/{repo}/pulls/{pr}/reviews/{review_id}

# Review に付いたインラインコメント
gh api repos/{owner}/{repo}/pulls/{pr}/reviews/{review_id}/comments

# 個別のディスカッションコメント (discussion_rxxx)
gh api repos/{owner}/{repo}/pulls/comments/{comment_id}

# Issue コメント (issuecomment-xxx)
gh api repos/{owner}/{repo}/issues/comments/{comment_id}
```

取得したら以下を記録する:
- **投稿者**: bot か人間か、どのツール (Copilot, Claude Assistant 等) か
- **投稿時刻**: どのコミット時点のコードを見たレビューかを特定するために重要
- **レビュー種別**: インラインコメント / レビュー本文 / issue コメント

## Phase 2: レビュー対象コードの特定

レビューがどの時点のコードを対象にしているかを特定する。これを間違えると検証結果が無意味になる。

```bash
# リモートの HEAD を確認
git log --oneline origin/{branch} -1

# PR のコミット履歴とタイムラインを確認
git log --oneline --format='%h %ai %s' {branch}
```

投稿時刻とコミットタイムラインを照合し、レビューが見たコードの時点を確定する。
修正コミットが push された後のレビューであれば、「修正後のコードを見た」のか「修正前の問題を自力で発見した」のかの区別が特に重要。

## Phase 2.5: PR スコープの確定

指摘がスコープ内かを判定するために、PR の変更範囲を先に確定する。

```bash
# PR の差分を取得
gh pr diff {pr_number} --patch

# PR の説明文・タイトルからスコープを把握
gh pr view {pr_number} --json title,body
```

### スコープ判定基準

| 判定 | 基準 |
|------|------|
| **スコープ内** | PR が新規作成・変更した記述やコードへの指摘 |
| **境界上** | PR が編集したセクション内の既存不備。同時修正は自然だが必須ではない |
| **スコープ外** | PR が触っていない既存の問題。別チケットで対応すべき |

### よくある誤判定パターン

- **「他にも同じ問題がある」→ スコープ外、は早計**: PR がその記述を新規作成しているなら、たとえ他の箇所にも同種の欠落があっても、当該箇所はスコープ内。「体系的欠落」を理由にスコープ外と判定してよいのは、PR が既存記述を触っていない場合のみ
- **スコープ外の指摘を「別チケット推奨」とする前に、既存チケットを検索する**: Jira 等で類似チケットが既にあれば「対応不要（既存チケットでカバー済み）」とする

### 集計時の注意

スコープ判定が「スコープ内」「境界上」「スコープ外」に分かれる場合、「N件中M件がスコープ内」のような集計は各カテゴリの件数を正確に数えること。境界上をスコープ内に含めて数えない。

## Phase 3: 検証可能な主張の抽出

レビューコメントから、コードに照らして検証可能な主張を列挙する。典型的な検証対象:

| カテゴリ | 例 |
|---------|-----|
| **行番号参照** | 「L360 に旧用語が残っている」 |
| **状態名・定数名** | 「PendingVerification は実装に存在しない」 |
| **用語の正確性** | 「旧用語が残存していない」 |
| **ファイル・シンボル参照** | 「auth-flow.ts で定義されている」 |
| **ロジックの整合性** | 「毎回必要」と「スキップ可能」の矛盾 |
| **網羅性の主張** | 「全10ファイルで統一されている」 |

## Phase 4: 各主張の検証

主張ごとに実際のコードを確認する。効率的な検証手段を選ぶ:

- **行番号参照の正確性**: `git show {commit}:{file} | sed -n '{line}p'` で当該コミット時点の内容を確認
- **状態名・シンボルの存在**: Serena の `find_symbol` や Grep で型定義・実装を検索
- **用語の残存確認**: Grep で対象ファイル全体を検索し、残存箇所を網羅的に列挙
- **ロジックの整合性**: 関連する複数箇所を読み、矛盾がないか確認
- **網羅性の検証**: 対象全ファイルを横断検索し、漏れがないかチェック

検証のポイント:
- レビューが参照するコミット時点のコードで確認する（現在の HEAD ではなく）
- 「正しい」だけでなく「他に同種の問題がないか」も確認する（レビューの網羅性評価）
- 断定表現（「残存していない」「すべて統一されている」）は特に厳密に検証する
- レビューがインターフェース越しの実装詳細（例: 「Repository の save がトランザクションか」）を主張している場合、差分に含まれないインフラ層の実装クラスまで読んで検証する。レビュー対象ファイルだけでは判定できないケースがある
