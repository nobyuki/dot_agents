# Review Comment Verifier

## 前提メモリ

スキル実行前に以下の Serena メモリを読み込むこと:

| メモリ名 | 用途 |
|---------|------|
| `project/plan_docs_location` | 計画書の保存先パス。必要に応じて公開状態を含む（設計判断の照合・検証根拠の客観性判断用） |

GitHub PR のレビューコメントに含まれる主張を、実際のコードとドキュメントに照らして検証し、構造化された妥当性評価を markdown で出力する。

## 評価の観点

- **事実の正確性**: 行番号参照、状態名、用語が実際のコードと一致するか
- **スコープ判定**: 指摘が当該 PR の変更範囲内か、境界上か、スコープ外か
- **問題発見力**: レビューが問題を自力で発見したのか、修正後のコードを見て「問題なし」と言っただけか
- **網羅性**: 指摘すべき類似箇所を漏れなくカバーしているか
- **断定の適切さ**: 主張の範囲が実際の検証範囲と一致しているか（過剰断定がないか）

## ワークフロー

1. **レビューコメントの取得** — URL パターンに応じて `gh api` で取得。投稿者・投稿時刻・種別を記録
2. **レビュー対象コードの特定** — 投稿時刻とコミットタイムラインを照合し、レビューが見たコードの時点を確定
3. **検証可能な主張の抽出** — 行番号参照、状態名、用語、ロジック整合性、網羅性の主張を列挙
4. **各主張の検証** — 当該コミット時点のコードと照合。断定表現は特に厳密に検証
5. **評価レポートの出力** — `/tmp/review-verify-{pr番号}.md` にファイルとして書き出し、パスをユーザーに伝える

## リファレンス

| ファイル | 内容 | 参照タイミング |
|---------|------|--------------|
| `references/workflow.md` | 各 Phase の詳細手順 | Phase 1-4: 検証の各段階 |
| `references/report-template.md` | レポートテンプレートと評価基準 | Phase 5: 評価レポート出力時 |
| `references/finding-numbering.md` | Finding ナンバリング規則（検証コンテキスト） | Phase 1: レビュー取得時の ID 採番、Phase 5: レポート出力時 |
| `references/pitfalls.md` | 過去の検証から得た知見・落とし穴 | 検証全体: 判断に迷ったとき |

## 注意事項

- 複数のレビューコメントが指定された場合は、それぞれ独立して検証したうえで総合評価にまとめる
- あるレビューが別のレビューの評価（メタレビュー）の場合は、元のレビューも含めて検証チェーンを追跡する
- レビューの「時点」を常に意識する。修正前を見たのか修正後を見たのかで評価が大きく変わる
- 日本語のレビューには日本語で、英語のレビューには英語で評価を出力する
- レビューの妥当性検証が終わった時点で、検証過程で得た知見（新たな検証パターン、見落としやすいポイントなど）があれば `references/pitfalls.md` に追記する
