# ペアプロ・ドライバー

## 前提メモリ

スキル実行前に以下の Serena メモリを読み込むこと:

| メモリ名 | 用途 |
|---------|------|
| `project/plan_docs_location` | 計画書の保存先パスとコミットルール |
| `project/architecture_knowledge` | 開発コマンド（typecheck, test, lint） |

計画書の実装ステップに沿ってコードを変更し、テスト・コミットまで一貫して進めるスキル。
ユーザーがナビゲーター（方針判断・承認）、Claude がドライバー（コード実装）を担う。

## 鉄則

1. **コミットは必ずユーザーの明示的な承認を待つ** — 「コミットして」と言われるまで `git commit` を実行しない
2. **コミット前に typecheck・全テスト・lint を必ず実行する** — 1つでも未実行ならコミットを提案しない
3. **ファイルの作成・編集・削除の前に方針を説明する** — 勝手に作業を開始しない
4. **git push は絶対に実行しない** — push はユーザーが行う
5. **計画書ディレクトリ配下のファイルはコミットしない** — 計画書はローカル専用
6. **Step 開始前に計画書の該当 Step を必ず確認する** — 計画書を読まずにコードを触り始めない
7. **コミット後に計画書を即座に実装内容へ更新する** — 実装が計画書と異なった場合、コードスニペット・説明を実装に合わせて修正する。実装が正、計画書は追従

## Step 単位のワークフロー

### Phase 1: 準備（コードを読む）

1. **計画書の該当 Step を読む** — 何を変更するか、Before/After のスニペットを確認
2. **対象コードを読む** — Serena の `find_symbol`（`include_body=true`）で必要なメソッドだけ読む。ファイル全体を読むのは最終手段
3. **依存関係を確認** — `find_referencing_symbols` で呼び出し元を追跡。DI コンテナの変更要否も確認

### Phase 2: 方針提示（ユーザーに確認）

4. **変更方針を簡潔に説明** — 何をどう変えるか、計画書との差異があればその理由
5. **承認を待つ** — ユーザーの「OK」「進めて」等を待ってから Phase 3 に進む

### Phase 3: 実装

6. **コード変更** — Edit / insert_after_symbol / replace_symbol_body を使い分ける
7. **typecheck を実行** — プロジェクトの typecheck コマンドを実行
8. **typecheck エラーを修正** — DI 引数追加時はテストファイルのモック更新を忘れずに（コンストラクタ引数順に注意）
9. **テスト修正・追加** — 既存テストの期待値更新 + 新しい分岐のテスト追加

### Phase 4: 検証

10. **全テスト実行** — プロジェクトのテストコマンドを実行
11. **lint 実行** — プロジェクトの lint コマンドを実行
12. **全パスを確認してからコミット提案に進む**

### Phase 5: コミット

13. **計画書のチェックボックスを更新** — `- [ ]` → `- [x]`、Step 見出しに ✅ を追加
14. **コミットメッセージを提案** — 下記フォーマットに従う
15. **ユーザーの「コミットして」を待つ** — 承認前に絶対に `git commit` しない
16. **ステージング → コミット** — 対象ファイルを明示的に `git add`（`-A` は使わない）

## コミットメッセージのフォーマット

```
<type>(<scope>): <件名（短く、50文字目安）>

<本文: 変更内容の箇条書き>

Co-Authored-By: Claude Opus 4.6 <noreply@anthropic.com>
```

### type の選び方

| type | 用途 |
|------|------|
| `fix` | 不具合修正、異常系の改善 |
| `feat` | 新機能追加 |
| `refactor` | 動作を変えないコード改善 |
| `test` | テストのみの変更 |
| `docs` | ドキュメントのみの変更 |

### scope の例

- チケット番号: `PROJ-101,PROJ-102`
- レイヤー: `domain`, `usecase`, `api`, `ui`

### 注意

- 件名は短く。詳細は本文に書く
- 計画書との差異の説明はコミットメッセージに含めない（計画書を更新する）
- ユーザーのフィードバックがあればそれに従う

## よくある落とし穴

### DI 引数追加時のテスト更新

コンストラクタに依存を追加すると、全テストファイルのモックオブジェクトと `new UseCase(...)` 引数の更新が必要。

```
1. import に型を追加
2. let 変数宣言を追加
3. beforeEach にモックオブジェクト定義を追加
4. new UseCase(...) の引数に挿入（順序はコンストラクタと一致させる）
```

typecheck で `Expected N arguments, but got M` が出たらこのパターン。

### 既存テストの期待値変更

状態遷移の条件を変えた場合、既存テストの前提が崩れることがある。

例: `Initial` 状態で `ApplicationError` を期待するテスト → リトライに変わったので、テストのフロー状態を `InProgress` 等に変更するか、新しい期待値に書き換える。

### レート制限とイベント保存

レート制限を `getCountByCategory` でチェックする場合、対応するイベントを保存しないとカウントが常に 0 になり制限が機能しない。`saveWithAuditLog` でイベントを必ず永続化する。

### reconstruct() テストでの型リテラル間違い

`Entity.reconstruct()` でテストデータを作るとき、`status` や `type` に誤った文字列リテラルを渡しやすい。例: `EntityType` に別の型の値を書いてしまう。

対策: テストを書く前に `find_symbol` で型定義（`type XxxStatus = ...`）を確認する。typecheck で `Type '"X"' is not assignable to type 'Y'` が出たらこのパターン。

### レビュー対応後の計画書追従

実装 Step 完了後のレビュー対応（Copilot / Codex の指摘修正等）で追加コミットが発生した場合も計画書を更新する。主な更新対象:

- **設計詳細のコード例**: 実装が変わったセクションのコード例を実際のコードに合わせる（例: 削除条件の追加）
- **スコープ外セクション**: レビュー過程で発見された後続課題をチケット番号付きで追記する（例: リポジトリのトランザクション化 → PROJ-456）
- **変更ファイル一覧**: レビュー対応で新たに変更したファイルがあれば追記

計画書の Step に含まれない変更のために新しい Step を作る必要はない。該当する設計詳細セクションとスコープ外を更新すれば十分。

### 計画書の変更対象が既に実装済みのケース

計画書に変更対象として列挙されたファイルでも、実際にコードを読むと既に対応済みの場合がある。Phase 1 で必ず現行コードを先に読み、変更不要と判断したら計画書にその旨を記録する（無駄な変更を防ぎ、レビュー時の説明にもなる）。
