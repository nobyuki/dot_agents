# ドキュメント更新計画書の作成

## 前提メモリ

スキル実行前に以下の Serena メモリを読み込むこと:

| メモリ名 | 用途 |
|---------|------|
| `project/plan_docs_location` | 計画書の保存先パスとコミットルール |
| `project/architecture_knowledge` | アーキテクチャ構成、ドキュメントディレクトリパス |
| `project/doc_survey_targets` | サーベイ対象ファイルリスト |

機能実装後に既存ドキュメントを最新の実装に合わせて更新するための計画書を作成する。
計画書は計画書ディレクトリに Markdown 形式で保存する。

## 計画書作成の原則

1. **全ドキュメントをサーベイする** — 影響範囲を漏れなく把握するため、ドキュメントディレクトリ全体と `CLAUDE.md` / `README.md` を調査する
2. **対象外を明示する** — 更新しないファイルとその理由を記載し、意図的な除外であることを示す
3. **Step 単位で作業を分割する** — 1 Step = 1 ファイル（または密接に関連する少数ファイル）で管理する
4. **変更箇所を具体的に示す** — 更新前/更新後の内容を明記し、レビュー可能にする
5. **実装コードと照合する** — ドキュメントの記載が実装と一致していることをコードで確認する
6. **コミットハッシュは記載しない** — 計画書はローカル専用であり、ハッシュは時間経過で意味を失う

## ワークフロー

1. **変更内容の把握** — 機能実装計画書や PR 差分から旧仕様→新仕様の対比を整理
2. **ドキュメントサーベイ** — 全ドキュメントをチェックし更新要否を判定。対象リストは `references/survey-targets.md` を参照
3. **計画書の作成** — テンプレートに従い記述。詳細は `references/structure-guide.md` を参照
4. **実装（ファイル単位で逐次コミット）** — 1 Step = 1 コミットが基本。計画書ディレクトリ配下は絶対にコミットしない
5. **スキルの更新** — 作成過程で得た知見を SKILL.md や references に反映。既存計画書参照テーブル（`references/structure-guide.md` 末尾）に追加

## 計画書の構造（概要）

```
# [チケット番号] ドキュメント更新計画書
## 1. 背景（何が実装されたか、旧仕様→新仕様の要点）
## 2. 更新対象ファイル一覧（# / ファイル / 優先度 / 更新内容 / 状態）
## 3. 更新詳細（ファイルごとのサブセクション）
## 4. 更新対象外（ファイル / 理由）
```

## ファイル名の規則

- チケット番号あり: `[チケット番号]-doc-update.md`（例: `PROJ-001-doc-update.md`）
- チケット番号なし: `doc-update-[description].md`（例: `doc-update-admin-invoice.md`）

## リファレンス

| ファイル | 内容 |
|---------|------|
| `references/structure-guide.md` | テンプレート詳細（各セクションの書き方、記載パターン A/B/C、既存計画書参照） |
| `references/survey-targets.md` | ドキュメントサーベイ対象の全リスト（優先度別） |
| `references/checklist.md` | 計画書完成前の品質チェックリスト |
