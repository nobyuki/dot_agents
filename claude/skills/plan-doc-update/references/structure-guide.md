# ドキュメント更新計画書 テンプレート詳細

## §1. 背景

実装の変更内容を簡潔にまとめる。

### 記載パターン

```markdown
## 1. 背景

[チケット番号] で [変更の概要] が実装された。
コード実装は完了しているが、既存ドキュメントが旧仕様のままであり、実装と乖離している。

### 変更の要点

- **旧仕様**: [旧仕様の説明]
- **新仕様**: [新仕様の説明]
  - [詳細1]
  - [詳細2]
```

## §2. 更新対象ファイル一覧

全更新対象をテーブルで一覧化する。

```markdown
| # | ファイル | 優先度 | 更新内容 | 状態 |
|---|---------|--------|----------|------|
| 1 | `path/to/file.md` | 高 | 更新内容の概要 | 未着手 |
| 2 | `path/to/file2.md` | 中 | 更新内容の概要 | 未着手 |
```

**優先度の基準**:
- **高**: API仕様、DB仕様、画面仕様など実装と直接対応するドキュメント
- **中**: フロー図、ユースケース、セキュリティなど間接的に影響するドキュメント
- **低**: 概要・ガイドなど参照頻度が低い、または影響が軽微なドキュメント

**状態の値**: `未着手` → `✅ 完了`

## §3. 更新詳細

ファイルごとにサブセクション（3.1, 3.2, ...）を設ける。
完了したセクションには `[完了]` プレフィックスを付ける。

### 記載パターン A: 簡潔な変更説明

小さな変更（1-2箇所の修正）の場合:

```markdown
### 3.1. [完了] tables.md — Entity テーブル

`external_service_id` と `created_at` の間に `mfa_enabled` カラムを追加。
```

### 記載パターン B: 複数箇所の変更

複数箇所を変更する場合:

```markdown
### 3.3. [完了] api-spec.md — エンティティ詳細取得・更新 API

- GET レスポンス: JSON サンプルとフィールド表に `mfaEnabled` を追加
- PUT: 説明文に MFA 必須設定を追加、リクエスト JSON・パラメータ表に追加
- `name` の必須を「○」→「-」に修正（`mfaEnabled` 単独更新が可能なため）
```

### 記載パターン C: before/after で示す大きな変更

大幅な変更の場合は Step 形式で before/after を明示:

```markdown
## Step 1: `ui-spec.md` — /usage UI 仕様更新

**対象ファイル:** `path/to/ui-spec.md`

### 1-1. 利用状況画面（L538-560）

現在:
[markdown block with current content]

更新後:
[markdown block with new content]

変更ポイント:
- ポイント1
- ポイント2
```

### Mermaid 図を含むドキュメントの更新

フロー図やシーケンス図（Mermaid 形式）を含むドキュメントでは、図中のステップ・分岐・ラベルの変更も計画に含める。変更箇所を before/after で示し、図全体ではなく差分のある部分を明記する。

## §4. 更新対象外

更新しないファイルとその理由をテーブルで示す。

```markdown
## 4. 更新対象外

| ファイル | 理由 |
|---------|------|
| `requirements.md` | ビジネス要件レベルの記述であり、実装詳細に依存しない |
| `roles.md` | 既存のパーミッションでカバー済み |
```

**よくある対象外理由**:
- ビジネス要件レベルであり実装詳細に依存しない
- 既存のパーミッション/設定でカバー済み
- 変更対象の機能に言及がない
- コード上のスキーマ定義で対応済み（OpenAPI 等）

## チェックリストの書き方

計画書の末尾にチェックリストを設ける。コミットハッシュは記載しない。

```markdown
## チェックリスト

- [ ] Step 1: `api-spec.md` に API セクション追加
- [ ] Step 2: `ui-spec.md` の画面仕様更新
- [ ] Step 3: `use-cases.md` にユースケース追加
```

完了時:
```markdown
- [x] Step 1: `api-spec.md` に API セクション追加
- [x] Step 2: `ui-spec.md` の画面仕様更新
```

## 既存計画書の参照

以下のタイプの既存計画書をスタイルの参考にできる:

| タイプ | 特徴 |
|--------|------|
| 標準的な構成 | 複数ファイル更新。簡潔な変更説明スタイル |
| Step 分割型 | 多数の Step 構成。ステップ内で行番号を参照し具体的 |
| before/after 詳細型 | before/after を詳細に記載するスタイル。API レスポンス例やテーブル差分が充実 |
