# レビュー時の落とし穴

過去のレビューセッションで実際に発生した問題から抽出したアンチパターン集。

## Copilot/Codex 指摘を invalid にする際の基準

PR 上の Copilot/Codex レビュー指摘を invalid と判定する前に、以下を確認する。

1. **コード上の事実として成立しているか**: 現時点で到達しないパスでも、コード上の論理として指摘が正しければ invalid ではない。重要度を下げる（Nit 等）のが適切
2. **「現時点では問題ない」は invalid の根拠にならない**: 将来の変更で顕在化するリスクを指摘している場合、invalid ではなく Nit / フォローアップ推奨に分類する
3. **OpenAPI 定義と実装の乖離**: 実装が正しく動作していても、OpenAPI 定義に記載がなければドキュメント整合性の問題として open（Nit）にする

| 判定 | 基準 | 例 |
|------|------|-----|
| invalid | 指摘の前提が事実と異なる | 「この関数は呼ばれない」→ 実際には呼ばれている |
| invalid | 対象外のコードへの指摘 | 本 PR で変更していないファイルへの指摘 |
| Nit（open） | 事実として成立するが、現時点で致命的ではない | P2002 の無条件キャッチ、OpenAPI 定義の欠落 |
| open | 事実として成立し、対応が必要 | セキュリティ上の問題、仕様不整合 |

## 再レビューで「対応済み」申告を検証する

PR 作者の対応コメントで「対応済み」と書かれている場合、以下を必ず確認する。

1. **指摘で求めたファイル/変更が実在するか**: 「テスト追加」と書かれていても、指摘したレイヤーのテストファイルが作成されているかを確認する。既存ファイルへのケース追加と新規ファイル作成は異なる対応
2. **対応コメントの記載と diff が一致するか**: 「X を修正しました」と書いてあっても、diff に X の変更が含まれていない場合がある
3. **部分対応を「対応済み」と表記していないか**: プラン検証のみ追加で「業務前提条件を検証」と書かれている場合、支払方法検証が欠落している可能性がある

特に AI ツールで対応している場合、AI が「対応しました」と自己申告した内容を PR 作者がそのまま転記しているケースがある。AI は「既存テストにケース追加」を「テスト追加」と報告するため、レイヤーの区別が落ちやすい。
