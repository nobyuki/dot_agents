# PR Review ワークフロー詳細

## Step 1: PR 情報の収集

PR URL からリポジトリと PR 番号を抽出し、以下を並列取得する:

```bash
gh api repos/{owner}/{repo}/pulls/{number}
gh api repos/{owner}/{repo}/pulls/{number}/reviews
gh api repos/{owner}/{repo}/pulls/{number}/comments
gh api repos/{owner}/{repo}/issues/{number}/comments
gh api repos/{owner}/{repo}/pulls/{number}/commits
```

対応するトピックブランチがローカルにチェックアウトされている場合は、ローカルファイルを直接読む。
diff の行番号ではなく、実ファイルの行番号を使うこと。

**gh コマンドの実行ポリシー**: 参照系の gh コマンド（`gh api` の GET リクエスト、`gh pr diff`、`gh pr view`、`gh pr list` など）はユーザーに確認せずそのまま実行してよい。書き込み系（`gh pr comment`、`gh pr review` など）のみ確認を取る。

## Step 2: 差分の取得と分析

差分の取得方法:
- ローカルにブランチがある場合: `git diff` またはファイル直接読み
- ない場合: `gh pr diff {number}`（大きい場合は `/tmp/` に保存）

分析の観点:
- **設計**: アーキテクチャパターンとの整合性、依存方向、責務分離
- **正確性**: ロジックの誤り、型の不整合、境界条件
- **セキュリティ**: インジェクション、認証・認可の漏れ
- **パフォーマンス**: N+1 クエリ、不要な全件取得
- **保守性**: 重複コード、命名、テストの有無
- **関連ファイルの整合性**: 機能削除・リネーム・ドキュメント修正の PR では、diff に含まれないファイルにも関連記述が残っていないかリポジトリ横断で検索する（例: 同じ機能を言及する別のドキュメント、削除されたシンボルの参照）

## Step 3: 再レビューの場合の追加分析

ユーザーが「再レビュー」「再度」「指摘が対応済みか確認」等の文言を使った場合、**過去の指摘事項の全件解消確認とデグレチェックを最優先**とする。新規 Findings の探索は二次的な位置づけとし、まず前回指摘の対応状況を網羅的に検証すること。

前回のレビューコメントが存在する場合:

1. **前回指摘の全件対応確認（最優先）**: 各指摘に対応するコード変更が入っているか、実ファイルで1件ずつ検証する。未対応・部分対応の指摘は明確にフラグを立てる
2. **デグレチェック（最優先）**: 前回以降の新規コミットで導入された変更が、既存の正常動作を壊していないか確認する。特に修正の副作用に注意
3. 対応コメント（PR 作者のレスポンス）の記載内容が実際の差分と一致するか検証
4. 「テスト追加済み」等の主張は、差分やリポジトリ内の検索で裏取りする

**関連PRのレビューコメント追跡**:

PR 説明文やチケットリンクから関連する先行PRが特定できる場合、そのPRのレビューコメントも取得する。
これにより以下を検証できる:
- 先行PRで確立されたパターン（例: `trim() || undefined`）が本PRでも一貫して適用されているか
- 先行PRで「別チケットで対応」と先送りされた指摘が本PRのスコープに含まれるか、または引き続き未対応か
- 先行PRのレビュー議論から得られた設計判断（意図的なスキップ等）が本PRでも踏襲されているか

```bash
# 関連PRのレビューコメントを取得
gh api repos/{owner}/{repo}/pulls/{related_number}/comments
```

## Step 3.5: PR 上のレビューコメント確認

`gh api repos/{owner}/{repo}/pulls/{number}/reviews` で取得した Copilot・Claude 等のボットレビューや他のレビュアーのコメントを確認する。

- 各指摘を実コードで検証し、対応済み・未対応・事実と異なるかを判定する
- 検証済みの指摘のうち、自身の Findings と重複するものは統合する
- 自身の Findings にない新たな有効指摘があればマージする
- レポートの「PR 上のレビューコメント確認」セクションに検証結果をテーブル形式で記載する
- ボットレビュー（Copilot 等）が複数ラウンド存在する場合、各ラウンドの指摘をすべて確認する。修正コミット後の再レビューで検出された指摘は、修正自体の副作用を捉えている可能性が高いため特に注意する。

**注意**: Codex 等ユーザーと一緒にレビューしているツールの指摘は「外部レビュー結果の検証」セクションとして分離しない。あくまで自身のレビューの Findings やフォローアップに統合する。PR 上に投稿されている Copilot/Claude 等のレビューのみ、別セクションで対応状況を記載する。

## Step 4: レポート出力

`references/templates.md` のテンプレートに従い、`/tmp/pr{number}-r{round}-review.md` にファイル出力する。
ラウンド番号は同一 PR に対するレビュー実施回数（初回=1, 再レビュー=2, ...）。
出力後、ファイルパスをユーザーに伝える。

**Finding ナンバリング（必須）**: レポート内のすべての Finding ID は `references/finding-numbering.md` の規則に従うこと。`Finding 0` や `Finding 1` のような連番は使用禁止。必ず `{レビュアー識別子}-R{ラウンド}.{通し番号}` 形式（例: `user1-R1.1`, `CP-R2.3`）を使う。見出し・本文参照・テーブルのすべてで統一すること。

**GitHub リンク化の回避**: レポート本文中で Finding 番号を参照する際、`#` を付けない（例: `user1-R1.1` であり `Finding #1` ではない）。GitHub 上では `#N` が issue/PR へのリンクに変換されるため、無関係なリンクが生成されてしまう。同様に `#issuecomment-` 等の参照もリンク化されるため、`issuecomment-` のように `#` を外す。

## プロジェクトコンテキストの活用

- CLAUDE.md にアーキテクチャやコーディング規約があれば参照し、プロジェクト固有の観点でレビューする
- DI パターン、レイヤー間の依存方向、命名規則など、プロジェクト固有のルールへの準拠を確認する
- 既存の類似コード（同種の seed スクリプト、同種の API 定義など）との一貫性を確認する

## スキルの継続的改善

レビュー中に得られた知見をこのスキル自体にフィードバックする。
改善の際は SKILL.md または `references/` 配下の該当ファイルを更新する。
