# レポートテンプレート

## セクション順序（必須）

レポートは以下の順序で構成する。読み手が最初に結論と要対応事項を把握できることを最優先とする。

1. **概要** — PR のメタ情報
2. **総合評価** — Approve / Request Changes の判定、Findings 一覧テーブル、Security follow-up、フォローアップ推奨
3. **Findings 詳細** — 各 Finding の詳細説明
4. **PR 上のレビューコメント確認**（該当がある場合） — Copilot/Claude 等ボットレビューの指摘検証
5. **設計書との整合性チェック**（設計書がある場合） — 設計書の記述と実装の対応確認
6. **事実検証結果** — 各 Finding の主張に対する cross-verify 結果
7. **前回指摘の対応確認**（再レビュー時のみ） — 前回レビューの指摘への対応状況
8. **デグレチェック** — 既存機能への影響確認
9. **まとめ** — 残存 Finding の要約と推奨アクション

## 初回レビュー

```
# PR #{number} レビュー結果

## 概要

**PR**: {title}
**ブランチ**: `{head}` → `{base}`
**変更**: {files}ファイル (+{additions}/-{deletions})
**コミット**: {commits}件

## 総合評価

{Approve / Request Changes}。{理由の要約}

### Findings

| ID | 重要度 | カテゴリ | 概要 |
|----|--------|----------|------|
| {ID} | {Must Fix / Should Fix / Nit} | {Process / Correctness / Security / Performance / Maintainability / Documentation} | {概要} |

（Nit を含む場合は「対応方針」列を追加する。分類フローは quality-standards.md「Nit の対応方針分類と Approve 整合性」を参照）

### Security follow-up (must create ticket)

（該当がある場合のみ。セキュリティ上の既存問題や、本 PR のスコープ外だが要チケット化の事項）

- {問題の概要。具体的なコード箇所と影響を明記}

### フォローアップ推奨

- **[Low]** {低優先度だが認識すべき事項}
- **[別スコープ]** {別 PR で対応すべき事項とチケットリンク}

## Findings 詳細

重要度ごとにテンプレートを使い分ける。AI が対応する場合に誤解しにくい構造（対象レイヤーの明示、非代替条件、箇条書きの完了条件）を優先する。

### Must Fix テンプレート

```
### {ID}: {タイトル} [Must Fix / {カテゴリ}]

{問題の事実。何が壊れている/要件を満たしていないかを1-2文で明示する}

これは `{対象レイヤー/責務}` に対する指摘です。現状のままでは `{不正な挙動 / 要件未達 / セキュリティ上の問題}` が残るため、この PR で解消が必要です。

未解消の論点:
- {論点1}
- {論点2}

完了条件:
- {必須条件1}
- {必須条件2}

修正先の第一候補: `{file}`
既存前例: `{file}`（あれば）
```

- 「代替されがちな対応」への非代替条件は、該当する場合のみ追加する（例: 「`{代替対応}` だけでは解消しません。」）。レイヤーが自明な場合など不要なら省略してよい
- Must Fix は Approve しない理由を明示する。完了条件は厳密に固定する

### Should Fix テンプレート

```
### {ID}: {タイトル} [Should Fix / {カテゴリ}]

{問題の事実。何が未整備/不十分かを1-2文で明示する}

これは `{対象レイヤー/責務}` に対する指摘です。現状でも直ちに破綻するわけではありませんが、`{回帰防止 / 保守性 / 契約整合性}` の観点から、この PR で合わせて対応すべきです。`{代替されがちな対応}` だけでは本指摘の確認範囲を満たしません。

未担保なのは以下です。
- {確認項目1}
- {確認項目2}

完了条件:
- {必要な変更/確認1}
- {必要な変更/確認2}

修正先の第一候補: `{file}`
既存前例: `{file}`（あれば）

{スコープ限定の補足（該当する場合のみ）。例: 本指摘は横並びの一般論ではなく、この PR で新規追加された {責務} に限定したものです。}
```

- 非代替条件（「`{X}` だけでは解消しません」）は、過去に代替対応で的を外されたケースや、AI が楽な方に流れやすい指摘では必ず書く
- レイヤーが自明な場合（例: Prisma スキーマの型不整合）は「対象レイヤー」の明示を省略してよい

### Nit テンプレート

```
### {ID}: {タイトル} [Nit / {カテゴリ}]

{改善したい点を1-2文で書く。今の実装でも動くが、どこにズレ/粗さがあるかを明示する}

これは `{対象レイヤー/責務}` の改善提案です。現時点で重大な不具合ではないものの、`{将来リスク / 整合性 / 保守性}` の観点で揃えておくと後続で扱いやすくなります。

提案:
- {改善案1}
- {改善案2}

修正先の第一候補: `{file}`
既存前例: `{file}`（あれば）
```

- Nit は対応任意であるため、非代替条件は書かない（Approve との矛盾を避ける）
- 「完了条件」ではなく「提案」ベースで記述する

### 共通の補足ルール

- **1 Finding 1論点**: 複数の要求を混ぜると、AI は修正コストの低いものだけ拾いやすい。別論点は別 Finding に分ける
- **該当箇所**: `{file}:{line-range}`（{何のコードか}）を記載する。Process 系など該当箇所がない場合は省略可
- **修正案（コード例）**: 必要な場合のみ記載する。テンプレートの構造化された完了条件で十分伝わる場合はコード例不要

#### Resolved Finding の表記

レビュー中に追加コミットで対応済みとなった Finding は、以下のパターンで記載する:

- Findings テーブル: `~~{概要}~~ → {N}th commit で resolved`
- 詳細セクションの見出し: `### {ID}: {タイトル} [{重要度} / {カテゴリ}] — resolved`
- 詳細セクション本文: 元の指摘を取り消し線で消し、対応内容を追記

```
### {ID}: {タイトル} [{重要度} / {カテゴリ}] — resolved

~~{元の指摘内容}~~

**{N}th commit ({sha}) で対応済み**: {対応内容の説明}
```

## 再レビュー（Nth round）

### セクション順序（再レビュー）

再レビューでも読み手が最初に結論と新規の要対応事項を把握できることを優先する。前回指摘の対応確認は「過去の指摘が解消されたか」の検証であり、新規 Findings の後に配置する。

1. **概要** — 前回レビューへのリンク、修正コミット一覧、変更ファイル
2. **総合評価** — Approve / Request Changes、Findings テーブル、フォローアップ推奨
3. **Findings 詳細** — 新規 Findings の詳細（なければ省略）
4. **前回指摘の対応確認** — 前回 Finding の対応状況テーブル + 各 Finding の対応検証
5. **PR 上のレビューコメント確認**（該当がある場合）
6. **事実検証結果**（該当がある場合）
7. **デグレチェック**
8. **まとめ**

```
# PR #{number} 再レビュー結果（{N}th round）

## 概要

前回レビュー（[issuecomment-{id}]({url})）の {前回Finding ID} に対する修正を検証。

| コミット | 内容 |
|---------|------|
| `{sha}` | {message} |

**変更ファイル**: {count}ファイル（{list}）

## 総合評価

**Approve / Request Changes**。{理由の要約}

### Findings

| ID | 重要度 | カテゴリ | 概要 |
|----|--------|----------|------|
| {ID} | {Must Fix / Should Fix / Nit} | {カテゴリ} | {概要} |

（Nit を含む場合は「対応方針」列を追加する。分類フローは quality-standards.md「Nit の対応方針分類と Approve 整合性」を参照）

（新規の指摘がなければ「新規 Findings なし」と明記）

### フォローアップ推奨

- ...

## Findings 詳細

（新規 Findings がある場合のみ。テンプレートは初回レビューの Must Fix / Should Fix / Nit を使い分ける）
