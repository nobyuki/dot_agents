# 品質基準

## 行番号の正確性

行番号は**必ず実ファイルで確認する**。diff のオフセットは累積でズレるため信用しない。

- ローカルにブランチがある場合: `Read` / `Grep` で実ファイルを読む
- ない場合: GitHub API でファイル内容を取得
- レポート出力前に、Findings テーブルと詳細セクションの全行番号・行レンジを実ファイルで再検証する。メソッド定義の行レンジは開始行（シグネチャ）から閉じブレースまでを含める

## ファイルパスの正確性

PR diff 対象のファイルパスを正確に特定してから読む。同名ファイルが複数パスに存在する場合、diff に含まれないファイルを誤参照するリスクがある。

- PR diff でファイルパスを確認してからローカルファイルを読む
- `glob` で同名ファイルが複数ヒットした場合、diff 対象のパスと一致するものを選ぶ
- 特に untracked ファイル（計画書ディレクトリ等のローカルコピー）と diff 対象ファイル（ルート直下等）の取り違えに注意

## 事実ベースの記述

推測表現を避け、検証可能な事実で記述する。

- NG: 「〜の経緯が読み取れる」「〜と思われる」
- OK: 「差分に含まれていない」「検索したが見つからなかった」「export が追加後に除去されている」

パフォーマンスや影響範囲の評価で実行コンテキスト（呼び出し元、実行頻度）に言及する場合は、実際の呼び出し元をコード検索で確認する。

## 対応状態の表現精度

指摘への対応状態は、実装の実態を正確に反映する表現を使う。

- NG: 「対応済（許容範囲）」（TODOコメントのみで未実装の場合）
- OK: 「未対応だが許容（TODOコメント留まり）」

| 実態 | 表現 |
|------|------|
| 修正コードがマージ可能な状態 | 対応済 |
| 代替策で軽減済み、本質的修正は未着手 | 部分対応（{代替策の内容}） |
| TODOコメントのみ | 未対応だが許容（TODO留まり） |
| 「別チケットで対応」宣言のみ | 未対応（別スコープ） |

- NG: 「全組織で定期実行されるため影響が大きい」（推測）
- OK: 「`CommitUsageEventsUseCase` から呼び出される（`usage-limit-alert-service.ts:22` のクラスコメント）。利用イベント commit のたびに実行される」

## 数値の扱い

変動しうる数値（ファイル数、import 箇所数など）は断定しない。

- NG: 「既存の import 箇所（6ファイル）」
- OK: 「主要な既存 import 箇所」

**実測必須の数値**: 以下はレポート出力直前に実測し、推測やキャッシュ値を使わない。

- PR の additions / deletions → `gh api repos/{owner}/{repo}/pulls/{number}` で取得
- テストケース数 → `grep -c '^\s*it(' {file}` で計測
- ファイル行数 → `wc -l {file}` で計測
- **commit 単位の変更ファイル数** → `gh api repos/{owner}/{repo}/commits/{sha}` で取得。PR 全体の diff からの推測ではなく、特定 commit に含まれるファイルを正確に確認する

レポート内で同じ数値を複数箇所に記載する場合（例: テスト行数をFinding詳細と対応確認の両方に書く場合）、全箇所を同一の実測値で統一する。

## テーブルの体裁

- セル内に改行を入れない（markdown テーブルが壊れる）
- 長い内容はテーブル外に記述するか、セルを分割する
- 出力前にテーブルのレンダリングが崩れないか確認する

## 表現の強度

- 「自動的に追従する」→「乖離が起きにくい構成」のように、過度に強い表現を避ける
- ただし Must Fix の根拠は明確かつ断定的に書く
- 未確定の設計方針に言及する場合、一方の選択肢に寄せた表現を避ける
  - NG: 「PR2 で 00:00 正規化に統一する」（どちらに確定するか未決定の場合）
  - OK: 「PR2 で方針確定後に設計書と JSDoc を一致させる」

## 事実と設計判断の区別

コード上の事実と、設計・運用上の判断を同じ強さで断定しない。

- NG: 「グローバルハンドラで処理されるため不要」（「不要」は設計判断）
- OK: 「グローバル `app.onError` が存在する。現行設計では個別 try-catch なしで動作する」

- NG: 「請求確定処理側で保証されるため問題なし」（本PR差分で検証不能）
- OK: 「PR作者は請求確定処理側で保証されるとの見解。本PR差分からは断定する根拠が不足」

特に「意図的にスキップ」の妥当性判断で、PR作者の見解をそのまま追認しやすいため注意。

## デグレチェックの記述精度

デグレチェック内の「問題なし」は、別の Finding と矛盾しないことを確認してから記載する。

- NG: テスト網羅性を「問題なし」としつつ、フォローアップで「自動テストがない」と記載
- OK: 「ユースケース側は追加済み。UI表示ロジックの自動テストは未整備」

既存モデルへのカラム追加・index 追加は「新規追加のみ」ではなく「既存定義の変更あり」として記載する。影響の有無は default 値・nullable 等の具体的な根拠と共に記述する。

- NG: 「新規追加のみで既存コードの変更はない」（既存モデルにカラム追加がある場合）
- OK: 「既存モデルにカラム追加（`fieldA` は default 0、`relatedEntityId` は nullable）。既存データに影響しない安全な変更」

「問題なし」は全面的に問題がないと読まれるため、部分的な場合は範囲を限定して記載する。

## 結論と Findings の整合性

結論（Approve / Request Changes）は Findings の重要度と矛盾してはならない。

- Should Fix 以上の未対応 Finding がある場合は **Request Changes** にする
- Approve にする場合は、残存 Finding がすべて Nit か、フォローアップチケットで対応済みであること
- 「Approve（条件付き）」は曖昧なため使わない。条件がある時点で Request Changes

## 列挙と箇所数の一致

修正箇所を具体的に列挙する場合、要約テーブルの「N箇所」と詳細リストの項目数を一致させる。
列挙後に箇所数を書く場合は、列挙を数えて記載する。先に箇所数を書いてから列挙する場合は、列挙完了後に数を再検証する。

- NG: 「5箇所を修正」と書きつつ7項目を列挙
- OK: 「7箇所を修正（金額フィールド6箇所の `?? 0` + taxRate の `?? 0.1`）」

## Finding の重要度分類

| レベル | 基準 | 例 |
|-------|------|-----|
| Must Fix | マージするとバグ・仕様不整合が発生する | API 定義とドメイン定義の乖離 |
| Should Fix | UX の劣化やメンテナンス性の問題 | 不明瞭なエラーメッセージ |
| Info / 別スコープ | 本 PR の範囲外だが認識すべき問題 | 関連するが別の機能のバグ |
| Nit | 対応任意の軽微な指摘 | コードスタイル、命名の微修正 |

**実運用で到達不可能なエッジケースの扱い**: 定数・ビジネスルール・上流バリデーションにより実運用で到達が防がれているエッジケースは、コード上の論理矛盾があっても Must Fix にしない。防御的プログラミングの提案として Nit に分類する。Must Fix は「現実の運用で発生しうるバグ」に限定する。

## Nit として報告しないもの

以下はノイズになるため Finding に含めない:

- **フィールド定義順序の差異**: OpenAPI スキーマと DTO 型定義のプロパティ順序が異なる程度の違い。動作に影響せず、対応不要と判断されることが多い
- **対応任意かつ説明だけで完結する指摘**: 報告しても PR コメントに残す必要がないレベルのものは、レポートに含めても価値が低い

## 修正コードの互換性検証（再レビュー時）

修正が入った場合、「指摘に対応しているか」だけでなく「修正自体が新たな問題を導入していないか」を検証する。

特に入力バリデーション追加時は要注意：
- 実際の呼び出し元（フロントエンド、他 API、テスト）が送信するフォーマットを確認する
- バリデーションスキーマが既存の全呼び出し元と互換性があるか検証する
- 例: Zod の `z.iso.date()` は `YYYY-MM-DD` のみ受け付けるが、フロントエンドが `toISOString()` で datetime を送る場合に非互換

修正→再修正の連鎖が起きた場合、最終形が元の指摘と全呼び出し元の両方を満たしているか通しで検証する。

## 新機能 PR のスコープ判断

新機能を追加する PR で、仕様書・ドキュメントに明記されたバリデーションが未実装の場合、「既存の類似機能も同じバリデーションがない」ことはスコープ外にする根拠にならない。新機能のコア仕様として本 PR のスコープに含めるべきか判断する際は、以下の観点を使う:

- **重大なビジネス影響に直結するか** — 未実装のまま本番稼働すると計算結果の破綻やデータ不整合が起きる場合はスコープ内
- **仕様に明記されているか** — ドキュメント等で値や制約が具体的に定義されている場合はスコープ内
- **既存パターンとのリスク差** — 既存機能と同じ未実装でも、新機能のほうがリスクが高い場合がある（例: 同じパラメータでも「ユーザーが自由に設定する値」vs「業務ルールで固定される値」では、後者のデフォルト値フォールバックが常に不正な結果を意味する）

スコープ判断で争点が生じた場合は、task-granularity-check スキルを参照し、「メインタスクに含める / 後続タスク候補に分ける」基準に照らして根拠を明確にする。

## バリデーション配置の推奨

修正案でバリデーション追加を提案する際は、ルールの性質に応じて配置場所を推奨する:

| ルールの性質 | 推奨配置 | 例 |
|------------|---------|-----|
| ビジネスルール（ドメイン知識） | ドメイン層（エンティティの `validate()` やドメインサービス） | パラメータの必須・値域・組み合わせ制約 |
| 入力形式・フォーマット | API 層（OpenAPI schema / リクエストバリデーション） | 文字列長、日付フォーマット |

API 層だけにバリデーションを置く提案は応急処置としては有効だが、別の入口（バッチ処理、内部 API 等）が増えた場合に抜ける。ドメイン層を最終防衛線とし、API 層で早期に 400 を返す二段構えを推奨する。

## Finding のスコープ一貫性

1つの Finding が複数の制約を含む場合（例: 「パラメータ必須化」+「許容値制約」）、Finding の概要・見出し・本文・フォローアップ参照・まとめの全箇所でスコープを統一する。一部だけ狭い表現にすると、レビュアーの意図（必須化だけ求めているのか、許容値まで含むのか）が読み手に伝わらない。

## Nit の対応方針分類と Approve 整合性

Nit を残したまま Approve する場合、対応方針の表現が Approve と矛盾しやすい。以下の決定フローで分類し、Nit を含む場合は Findings テーブルに「対応方針」列を追加して記載する（Nit がなければ列は不要）。

```
non-blocking 指摘
├─ 質問・確認系 → 回答必須（後続コメントは再判定）
├─ 妥当性が低い → 対応不要
└─ 妥当性が高い
   ├─ 既存チケットあり → 既存チケットにコメント追記 + PRにリンク残してスレッド解決
   └─ 既存チケットなし
      ├─ YAGNI → 必要に応じてコードコメントで補足、JIRA 起票しない
      └─ not YAGNI
         ├─ 対応コスト低 → 同PRで対応「推奨」
         └─ 対応コスト高 → JIRA 起票（H/M/L）
```

**YAGNI の判定基準**: 「将来必要になるかもしれない」改善は YAGNI。「現時点で事実と乖離している」（例: コメントやドキュメントが実装と不一致）は not YAGNI。

**「推奨」の付与（必須）**: Nit + Approve の組み合わせでは、対応方針に必ず「推奨」を付ける。「同PRで対応」と断定すると Approve と矛盾して見える。「同PRで対応推奨」なら Nit（対応任意）と整合する。

## テスト件数の対象明示

テスト件数を複数箇所で言及する場合、対象ファイル/レイヤーを明示する。同じ「UseCase テスト」でも複数ファイルにまたがる場合、合計値と個別値が混在して読み手を混乱させる。

- NG: 「UseCase テストは全 5件」（別ファイルに 2件あり合計 7件の場合）
- OK: 「create の UseCase テストは全 5件（get-status の 2件は別ファイル）」

## GitHub スレッド状態と実装完了の区別

PR 上のレビューコメントが resolved（スレッド閉じ）されていても、実装が完了しているとは限らない。Status 列の値は `finding-numbering.md` で定義された `open / resolved / merged / invalid` を使い、GitHub スレッド状態との乖離は確認箇所欄で補足する。

| 状態 | Status | 確認箇所欄の補足 |
|------|--------|----------------|
| スレッド閉じ + 実装完了 | `resolved` | （不要） |
| スレッド閉じ + 実装未完了 | `open` | 「GitHub スレッドは resolved だが実装未完了」 |
| スレッド開 + 実装未完了 | `open` | （不要） |

再レビュー時に前回指摘の対応テーブルで open としている項目が、PR 上のレビューコメント確認テーブルで resolved になっていると内部矛盾に見える。確認箇所欄の補足で乖離の理由を明記する。

## 全称表現の回避

「全て」「全件」「全ファイル」等の全称表現は、反例が1つでも存在すると主張が崩れる。実測で全数確認していない場合は使わない。

- NG: 「同階層の他 API は全てテストファイルが整備されている」（未テスト API が複数ある場合）
- OK: 「同階層に API テストの前例が複数ある（`billing-api.test.ts`, `invoices-api.test.ts` 等）」

## AI 誤読を潰すレビューコメントのルール

PR 作者が AI ツールにレビューコメントをそのまま読ませて対応する場合、AI が指摘の意図を取り違えるリスクがある。以下のルールで解釈の余地を潰す。

1. **1 Finding 1論点にする**: 複数の要求を1つに詰めると、AI は修正コストの低いものだけ拾いやすい
2. **対象レイヤー/責務を明示する**: 最初に「これは API ハンドラの責務に対する指摘です」のようにスコープを固定する。レイヤーが自明な場合は省略可
3. **何では解消しないかを書く**: AI は近い修正で済ませようとするため、非代替条件を先に潰す。例: 「UseCase テスト追加のみでは本指摘の確認範囲を満たしません」。過去に代替対応で的を外されたケースでは必須
4. **完了条件を箇条書きで列挙する**: AI は自然文の含意より箇条書きの条件を拾いやすい

※ 修正先ファイル・既存前例・PR 固有スコープの明示は Finding テンプレート（`references/templates.md`）に組み込み済み

**避けるべき表現**（人間には通じるが AI には抽象度が高すぎる）:
- 「テストを追加してください」（何のテストか不明）
- 「API 層もカバーした方がいいです」（義務か提案か不明）
- 「最低限の回帰テストが必要です」（何が最低限か不明）
- 「このままだと将来壊れます」（何をすべきか不明）

## 他ツールのレビュー結果のマージ

Copilot や Codex のレビュー結果が提示された場合:

1. **各指摘を実コードで検証する**（そのまま信用しない）
2. 行番号が不正確な場合は実ファイルで修正する
3. 検証済みの指摘を自身の Findings にマージし、重複を統合する
4. 検証結果に差異がある場合はその旨を明記する
5. 検証済みの指摘を Findings に統合する際、Finding 詳細に「検出元: {ツール名}レビューで指摘、実コードで検証済み」を1行付記する（独立セクションは不要）
6. **Finding ID は `references/finding-numbering.md` の規則に従う**。ユーザーがペーストした外部レビュー結果に独自の番号体系（例: `1.`, `Finding-1`）が使われていても、レポートに記載する際は必ず正規フォーマット（例: `CX-R1.1`）に変換する

## 「対応前に確認必須」の記述精度

Finding に「対応前に確認必須」を記載した場合、参照先のリポジトリルート相対パスと確認すべき内容を具体的に書く。
参照先を列挙するだけだと AI が確認対象を特定できず読み飛ばしやすい。指示形で何を確認するかを明記する。

- NG: 「参照: billing-master-spec.md」（確認対象が曖昧）
- OK: 「`docs/docs/for-developers/billing-master-spec.md` を読み、支払方法の種別定義（credit_card / bank_transfer の区分）を確認すること」

記載した参照が Finding の根拠またはスコープ判断にどう効いたかを本文で結びつける。
