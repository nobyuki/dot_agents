# PR Review Skill

## 前提メモリ

スキル実行前に以下の Serena メモリを読み込むこと:

| メモリ名 | 用途 |
|---------|------|
| `project/plan_docs_location` | 計画書の保存先パス。必要に応じて公開状態を含む（設計判断の照合用） |

GitHub PR のコードレビューを実施し、構造化された markdown レポートを出力する。

## ワークフロー

1. **Step 1: PR 情報の収集** — PR URL から gh api で PR 本体・レビュー・コメント・コミットを並列取得
2. **Step 2: 差分の取得と分析** — 設計・正確性・セキュリティ・パフォーマンス・保守性・関連ファイル整合性の観点で分析
3. **Step 3: 再レビューの追加分析** — 前回指摘の全件対応確認とデグレチェックを最優先で実施
4. **Step 3.5: PR 上のレビューコメント確認** — Copilot 等のボットレビューや他レビュアーの指摘を検証・統合
5. **Step 4: レポート出力** — テンプレートに従い `/tmp/pr{number}-r{round}-review.md` に出力

各ステップの詳細手順は `references/workflow.md` を参照。

## リファレンス

| ファイル | 内容 | 参照タイミング |
|---------|------|--------------|
| `references/workflow.md` | ワークフロー詳細手順（コマンド例・分析観点・再レビュー手順） | Step 1-4: 各ステップの実行時 |
| `references/templates.md` | レポートテンプレート | Step 4: レポート出力時 |
| `references/quality-standards.md` | 品質基準・重要度分類・他ツールマージ | Step 2: 差分分析時 |
| `references/finding-numbering.md` | Finding ナンバリング規則 | Step 4: レポート出力時 |
| `references/pitfalls.md` | レビュー時の落とし穴（invalid 判定基準・対応済み申告の検証） | Step 3: 再レビュー時、Step 3.5: ボットレビュー検証時 |
