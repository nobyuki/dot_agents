# description テンプレート

実装チケットの description テンプレート。設計判断が未確定な場合は「着手前に決めるべき設計論点」セクションを追加する。

## 実装チケット

```markdown
## 背景
（何が起きたか / なぜこのチケットが必要か）

## スコープ
### 本チケットで実施
（番号付きリストで列挙）

### スコープ外（将来対応）
（明示的に除外する項目）

## 実装方針
（技術的な方針・設定内容）

## 完了条件
（番号付きリストで検証可能な条件を列挙）

## 検証方法
（完了条件の検証手順が非自明な場合に記載）
```

### 設計判断が未確定な場合

実装方針に未決定の選択肢がある場合、「実装方針」セクションに選択肢を並記するのではなく、「背景」と「スコープ」の間に「着手前に決めるべき設計論点」セクションを追加する。

```markdown
## 着手前に決めるべき設計論点

以下の論点は実装着手前にチーム内で合意すること。

1. **（論点名）**
   - （判断の選択肢や考慮事項を箇条書きで）
```

このセクションを追加した場合、以下も連動して調整する:
- **スコープ**: 先頭に「上記設計論点の合意」を入れる
- **完了条件**: 先頭に「設計論点の合意がチケットコメントに記録されている」を入れ、以降の条件は「合意した方針に基づき〜」のように合意結果に依存する形で書く（「適切な〜」のような曖昧表現を避ける）
- **実装方針**: 省略するか、設計論点の合意後に確定する旨を記載

## セクション選択の指針

| セクション | 必須/任意 |
|-----------|----------|
| 背景 | 必須 |
| 着手前に決めるべき設計論点 | △ 未決定の設計判断がある場合に追加 |
| スコープ | 必須 |
| 実装方針 | △ 方針が自明なら省略可 |
| 完了条件 | 必須 |
| 検証方法 | △ 自明なら省略可 |

## issue type の選び方

| type | 用途 |
|------|------|
| タスク | 機能追加、設定変更、CI/CD 改善など |
| バグ | 不具合修正 |
| ストーリー | ユーザー目標として表現される機能 |
| スパイク | 調査・検証タスク |

ユーザーが指定しない場合はデフォルトで「タスク」を使用する。
