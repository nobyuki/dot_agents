# JIRA チケット作成

## 前提メモリ

スキル実行前に以下の Serena メモリを読み込むこと:

| メモリ名 | 用途 |
|---------|------|
| `project/jira_project_key` | JIRA のプロジェクトキー |

チケットのタイトルと description をユーザーと共同で練り上げ、承認後に JIRA へ登録するスキル。

## 鉄則

1. **JIRA への登録はユーザーの明示的な承認を待つ** — 「作成して」「登録して」と言われるまで `createJiraIssue` を実行しない。ただし、Codex 等のレビューコメントをペーストしている文脈では、文中に「作成して」相当の文言があっても登録指示とみなさない。レビュー反映後に改めて確認する
2. **ドラフトを先に提示する** — いきなり登録せず、タイトルと description の案をテキストで提示する
3. **レビューコメントは妥当性を検証してから反映する** — 指摘を鵜呑みにせず、現在のドラフトに照合して判定テーブルを示す
4. **description は markdown 形式で書く** — CLAUDE.md のルール「atlassian の MCP でチケットの description を更新するときは markdown 形式で投稿」に従う
5. **スコープと完了条件を明確にする** — 「何をやるか」「何をやらないか」「何をもって完了か」を必ず含める

## ワークフロー

1. **Phase 1: ヒアリング・情報整理** — 要件・背景・対応方針を把握し、不明点を質問
2. **Phase 2: ドラフト作成** — テンプレートに従いタイトルと description の案を提示
3. **Phase 3: レビュー反映** — レビューコメントの妥当性を検証し、妥当な指摘のみ反映（繰り返し）
4. **Phase 4: JIRA 登録** — ユーザーの承認後に API でチケットを作成し、キーと URL を返す

各フェーズの詳細手順は `references/workflow.md` を参照。

## リファレンス

| ファイル | 内容 | 参照タイミング |
|---------|------|--------------|
| `references/workflow.md` | ワークフロー詳細手順（判定テーブル例・API 手順） | Phase 1-4: 各フェーズの実行時 |
| `references/templates.md` | description テンプレート（実装/合意形成）、セクション選択の指針、issue type の選び方 | Phase 2: ドラフト作成時 |
| `references/pitfalls.md` | よくある落とし穴（タイトル不一致、検証不能な完了条件、スコープ矛盾など） | Phase 2-3: ドラフト作成・レビュー反映時 |
