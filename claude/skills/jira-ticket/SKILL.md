# JIRA チケット作成

## 前提メモリ

スキル実行前に以下の Serena メモリを読み込むこと:

| メモリ名 | 用途 |
|---------|------|
| `project/jira_project_key` | JIRA のプロジェクトキー |

チケットのタイトルと description をユーザーと共同で練り上げ、承認後に JIRA へ登録するスキル。

## 鉄則

1. **JIRA への登録はユーザーの明示的な承認を待つ** — 「作成して」「登録して」と言われるまで `createJiraIssue` を実行しない。ただし、Codex 等のレビューコメントをペーストしている文脈では、文中に「作成して」相当の文言があっても登録指示とみなさない。レビュー反映後に改めて確認する
2. **ドラフトを先に提示する** — いきなり登録せず、タイトルと description の案をテキストで提示する
3. **レビューコメントは妥当性を検証してから反映する** — 指摘を鵜呑みにせず、現在のドラフトに照合して判定テーブルを示す
4. **description は markdown 形式で書く** — CLAUDE.md のルール「atlassian の MCP でチケットの description を更新するときは markdown 形式で投稿」に従う
5. **スコープと完了条件を明確にする** — 「何をやるか」「何をやらないか」「何をもって完了か」を必ず含める

## ワークフロー

### Phase 1: ヒアリング・情報整理

1. **ユーザーの要件を整理** — 何を解決したいか、背景・原因・対応方針を把握
2. **不明点があれば質問** — スコープ、優先度、issue type 等が不明なら確認

### Phase 2: ドラフト作成

3. **タイトルと description の案を提示** — 下記テンプレートに従い、markdown コードブロックで表示
4. **ユーザーのフィードバックを待つ** — 修正指示・レビューコメントがあれば Phase 3 へ

### Phase 3: レビュー反映（繰り返し）

5. **レビューコメントの妥当性を検証** — 現在のドラフトに照合し、判定テーブルを提示

   | # | 指摘 | 判定 | 根拠 |
   |---|------|------|------|
   | 1 | 指摘内容 | ✅ 対応済み / ✅ 妥当 / ⚠️ 部分的 / ❌ 不要 | 根拠 |

6. **妥当な指摘のみ反映してドラフトを更新** — 更新後の全文を再表示
7. **Phase 2-3 をユーザーが満足するまで繰り返す**

### Phase 4: JIRA 登録

8. **ユーザーの「作成して」「登録して」を待つ** — 承認前に絶対に API を呼ばない
9. **プロジェクト情報を取得** — `getAccessibleAtlassianResources` で cloudId、`getJiraProjectIssueTypesMetadata` で issue type を取得
10. **チケットを作成** — `createJiraIssue` で登録し、チケットキーと URL を返す

## description テンプレート

チケットの性質に応じてテンプレートを使い分ける。

### 実装チケット

```markdown
## 背景
（何が起きたか / なぜこのチケットが必要か）

## スコープ
### 本チケットで実施
（番号付きリストで列挙）

### スコープ外（将来対応）
（明示的に除外する項目）

## 実装方針
（技術的な方針・設定内容）

## 完了条件
（番号付きリストで検証可能な条件を列挙）

## 検証方法
（完了条件の検証手順が非自明な場合に記載）
```

### 合意形成チケット

実装ではなく意思決定が目的のチケット（方針決定、設計判断など）。

```markdown
## 背景
（現状と、方針決定が必要な理由）

## 論点
（番号付きリストで判断すべきポイントを列挙）

## 選択肢
### A: （選択肢名）
（説明・実装イメージ）

### B: （選択肢名）
（説明・実装イメージ）

### C: （選択肢名）
（説明・実装イメージ）

## 完了条件
（採用案・選定理由の記録、後続チケットの起票など）
```

### セクション選択の指針

| セクション | 実装チケット | 合意形成チケット |
|-----------|------------|----------------|
| 背景 | ✅ 必須 | ✅ 必須 |
| スコープ | ✅ 必須 | - |
| 論点 | - | ✅ 必須 |
| 選択肢 | - | ✅ 必須 |
| 実装方針 | △ 方針が自明なら省略可 | - |
| 完了条件 | ✅ 必須 | ✅ 必須 |
| 検証方法 | △ 自明なら省略可 | - |

## issue type の選び方

| type | 用途 |
|------|------|
| タスク | 機能追加、設定変更、CI/CD 改善など |
| バグ | 不具合修正 |
| ストーリー | ユーザー目標として表現される機能 |
| スパイク | 調査・検証タスク |

ユーザーが指定しない場合はデフォルトで「タスク」を使用する。

## よくある落とし穴

### タイトルとスコープの不一致

タイトルが示す範囲と、description 内のスコープ・完了条件が一致しているか確認する。
スコープ外セクションに書いた内容が背景や完了条件に混在していると、実施範囲が曖昧になる。
特に、スコープ修正後にタイトルを再検証すること。スコープを広げた/狭めたのにタイトルが元のままだと、修正しても不一致が残る。

### 完了条件に検証不能な項目

「〜を追加する」は完了条件ではなく作業内容。完了条件は「〜が動作する」「〜が不可になる」のように検証可能な状態で書く。

### 検証手順の前提条件の省略

検証に特別な権限（管理者権限等）や環境（検証用ブランチ等）が必要な場合、明記しないと実施者が詰まる。

### 背景の原因表現の断定

複合要因の場合、「原因は〜」と単一原因に断定せず「主要因は〜」とする。監査・振り返り文書として過度な単一原因化を避ける。

### 目的とスコープの矛盾

ドラフト完成時に「チケットの目的を達成するために必要な手段がすべてスコープに含まれているか」を検証する。目的達成に必須の手段をスコープ外に置くと、チケットを完了しても目的が達成されない。

例: 「マージ後の型破綻を検知したい」が目的なのに、up-to-date 強制（Require branches to be up to date before merging）をスコープ外にすると、PR 単体では通るがマージ後に壊れるケースを防げない。

### 実装制約の見落とし

スコープに「メソッドをトランザクションでラップする」と書いても、内部で呼んでいるプライベートメソッドが DB クライアントを直接使っていると、そのままではトランザクションに参加できない。スコープに書く際は、対象メソッドの内部構造を確認し、前提となる変更（例: 「内部メソッドがトランザクションコンテキストを受け取れる構造に変更」）も明記する。

### テスト手法・判定基準の曖昧さ

「失敗注入テストを追加」だけでは実行可能性が低い。テスト手法（例: 「Prisma モックで2段目の操作を失敗させる」）と判定基準（例: 「1段目の永続化が残らないことをDB状態で確認」）を具体的に書く。

### 異なる特性を持つ対象の一括記述

1つのチケットで複数対象を扱う場合、特性が異なる対象（例: 単純な2操作 vs 暗号化処理を含む複雑な操作）は個別に振る舞いの方針を明記する。「全件原子化」と読める記述だと、現行の部分成功仕様を維持すべき箇所まで変更対象に見える。

### 検証手順の時系列整合性

検証手順を書いたら、手順を上から順にシミュレーションして各ステップの前提が成立するか確認する。特に「ある状態を作ってから別の操作で壊す」再現手順では、操作の順序を間違えると期待する状態が作れない。

例: 「PR 単体は緑、ベース更新後に赤」を再現するには、モック未追随 PR を先に作って緑を確認し、その後にインターフェース拡張をベースにマージする。逆順だと PR 作成時点で赤になり再現できない。

### PRレビュー指摘からのチケット起票

PR レビューの指摘事項をチケット化する場合、まず既存チケットとの関係を評価する:

- **案 A: 既存チケットにコメント追加** — 指摘の関心事が既存チケットのスコープと重なる場合（例: 同じファイルを触る作業のついでに対応可能）
- **案 B: 新規チケットとして起票** — 関心事が独立している場合（例: トランザクション保証 vs コード重複解消）

判断基準をユーザーに提示し、どちらにするか選択してもらう。

### 既存チケットとの重複・部分重複

修正方針を複数挙げるチケットでは、一部が別チケットや進行中 PR で既に対応済みのことがある。
ドラフト作成時にコードベースと既存チケットを照合し、部分重複がある場合は：
- どの方針が対応済みか（チケット番号・PR 番号付きで明記）
- 残りの方針だけで本チケットのスコープとして成立するか
を整理する。放置すると担当者が重複作業を行うか、対応済み部分との矛盾で混乱する。

### 実装方針と完了条件の網羅性検証

実装方針にコード例を含める場合、そのコードが完了条件の全項目を満たすかシミュレーションする。
特に「入力値の種類」×「期待する出力」のマトリクスを作り、コード例がすべての組み合わせで正しく動作するか確認する。

### 技術的前提の連鎖確認

ある値の変更が効果を発揮するには、上流・下流の前提が揃っている必要がある場合がある。
スコープに書く修正の前後にボトルネックがないか確認する。

例: トークン A（10分）→ トークン B（5分→60分に延長）の順で発行される場合、
B を延ばしても A がボトルネックになる。
