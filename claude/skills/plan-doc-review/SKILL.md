# 実装計画書レビュー

## 前提メモリ

スキル実行前に以下の Serena メモリを読み込むこと:

| メモリ名 | 用途 |
|---------|------|
| `project/plan_docs_location` | 計画書の保存先パスとコミットルール |

計画書を「実装前の設計レビュー」として扱い、重大な欠陥を先に潰す。
要約よりも、実装時に事故になる具体的な指摘を優先する。

## レビュー手順

1. 計画書を読む
- 対象ファイルを全文確認する。
- チケット番号、対象ユースケース、変更レイヤー（domain/application/infrastructure/interface）を抽出する。

2. 現行コードと突き合わせる
- 計画書内で変更対象に挙がっているファイルを実際に開く。
- ステータス遷移、例外処理順序、永続化タイミング、冪等性の前提が現行コードと一致するか確認する。
- 既存の再送・レート制限・エラーハンドリングの実装を確認し、計画との差分を明確化する。

3. 重大度で判定する
- `Critical`: 仕様を満たさない、または本番障害になり得る設計ミス
- `High`: 回帰やセキュリティ/整合性リスクが高い設計漏れ
- `Medium`: 実装時に破綻しやすい不整合、テスト不足、曖昧仕様
- `Low`: 可読性や運用性の改善提案

4. 指摘を根拠付きで出す
- 各指摘に以下を含める。
- 何が問題か
- なぜ問題か（挙動・失敗モード）
- 証拠（計画書の行番号 + 実コードのファイル/行）
- 最小修正案

5. テスト観点を補完する
- 正常系だけでなく、再実行・失敗後リトライ・期限切れ・競合更新を含む回帰テストを要求する。
- 既存テストで不足するケースを明記する。

6. レビュー完了後にスキルを更新する
- レビュー中に見つかった新しい観点、見落としやすい失敗パターン、repo 固有ルールをこの `SKILL.md` に反映する。
- 次回以降のレビュー品質を上げるため、チェックリストや重大度判定基準を必要に応じて改善する。

## 出力フォーマット

- `指摘事項（重大度順）`
- `補足（方針自体）`
- 指摘がゼロの場合は `重大な指摘なし` を明示し、残存リスクと追加テストだけ記載する。

## チェックリスト

- [ ] 計画書の前提が現行コードと一致している
- [ ] 例外処理の順序で取りこぼしがない
- [ ] 冪等性が API 再実行シナリオで成立する
- [ ] データ整合性（unique 制約・削除カスケード）に矛盾がない
- [ ] レート制限と悪用耐性が担保されている
- [ ] 追加/修正テストが十分に定義されている

## repo 固有ルール

- 計画書ディレクトリはローカル計画用でコミット対象外。レビュー時に「未コミット」を問題として指摘しない。
