# ストーリーポイント概算ガイド

## SP基準テーブル

本スキルで使用する固定の概算基準。1 SP ≈ 半日を目安とする。

| SP | 規模感 | 典型的な作業 |
|---:|-------|-----------|
| 1 | 極小 | 設定値変更、確認作業、方針決定済みの完了記録 |
| 2 | 小 | 小規模バグ修正、ドキュメント整理、調査スパイク、単一ファイルの修正 |
| 3 | 小〜中 | 中規模バグ修正、単一レイヤーの機能追加（APIのみ or UIのみ）、cherry-pick/バックポート |
| 5 | 中 | API+UI の機能追加、複数ユースケースの修正、環境構築、外部サービス連携 |
| 8 | 大 | 複数ドメインにまたがる機能実装、ドメインロジック変更+テスト、分割推奨級の大型タスク |
| 13 | 特大 | ストーリー級（複数PR分割前提）、基盤変更+全レイヤー影響 |

## 見積もりの判断基準

### 加算要因（+1〜2 SP）

- 外部サービス連携（決済API、外部SaaS等）が含まれる
- DBマイグレーションを伴う
- 既存テストの大量修正が必要
- 影響範囲の調査が必要（原因不明のバグ等）
- セキュリティ・課金に関わるロジック（慎重なレビュー・テストが必要）

### 減算要因（-1 SP）

- 既存の類似実装（参考実装）がある
- 変更が機械的（エラーメッセージ変更、設定値追加等）
- 既にレビュー済みのコードのバックポート

### 見積もり不能（? SP）

以下の場合は `?` とし、合計SPから除外する:
- description が空でスコープが定義されていない
- 「見直し」「整理」など範囲が曖昧なチケット
- 調査結果次第で工数が大きく変動するチケット（調査スパイクは別途見積もり可能）

## 集計ルール

1. **親ストーリー**: SP は「—」と記載。子タスクの合算値を備考に `実工数は{子KEY}({SP})+{子KEY}({SP})={合計}` の形式で記載
2. **見積もり不能チケット**: 合計から除外し、除外した旨を注記
3. **消化率**: `完了合計 / (未完了合計 + 完了合計) * 100` で算出（小数点以下切り捨て）
4. **根拠列**: 1行で具体的な作業内容と判断理由を記載（例: 「API+UI+テスト。中規模機能追加」）

## description に工数見積もりが記載されている場合

チケットの description に「推定工数: X日」等の記載がある場合:
- 参考値として活用する（そのまま採用するのではなく、SP基準テーブルと照合して妥当性を検証）
- 根拠列に「descriptionにX日と記載」と注記
- 換算: 1日 ≈ 2 SP
